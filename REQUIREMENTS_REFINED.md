# 日志异常分诊工具 - 需求文档（精炼版）

## 📋 项目概述

### 项目目标
构建一个轻量级日志异常分诊 MVP，核心价值是：
1. **自动识别**混乱日志中的异常
2. **智能合并**同一根因的重复异常
3. **优先级排序**指导处理顺序
4. **提供建议**区分自动处理和人工介入
5. **可解释性**所有判断都能追溯到明确规则

### 评估重点（按优先级排序）
1. ⭐⭐⭐ **独立判断能力** - 模糊需求下的拆解和决策
2. ⭐⭐⭐ **AI 协作能力** - 驾驭 AI 而非依赖 AI
3. ⭐⭐⭐ **可解释性** - 每个决策都有明确推理过程
4. ⭐⭐ **数据建模能力** - 处理脏数据和边界情况
5. ⭐⭐ **测试意识** - 测试核心逻辑而非表面功能
6. ⭐ **工程质量** - 代码结构、文档完整性

## 🎯 核心功能需求

### 1. 日志读取与容错
**输入：** JSONL 格式日志文件（至少 30 条）  
**处理：**
- 解析有效 JSON 行
- 识别无效行并记录为脏数据
- 无效行不导致程序崩溃
- 区分有效日志和脏数据

**成功标准：**
- ✅ 能读取混合有效/无效行的文件
- ✅ 脏数据被记录但不中断流程
- ✅ 输出明确区分有效和无效数据

---

### 2. 字段标准化
**输入：** 字段名不一致的原始日志  
**输出：** 统一的内部模型

**标准化字段：**
```python
{
    "id": str,              # 唯一标识
    "timestamp": datetime,  # 统一时间格式
    "source": str,          # 标准化来源
    "level": str,           # 统一日志等级
    "message": str,         # 错误信息
    "job_name": str,        # 任务名称（可选）
    "trace_id": str,        # 追踪ID（可选）
    "raw": dict,            # 原始数据
    "is_dirty": bool,       # 是否脏数据
    "dirty_reason": str     # 脏数据原因
}
```

**处理策略：**
- 时间字段：timestamp/time/created_at → timestamp
- 服务字段：service/source/app/system → source
- 等级字段：level/severity/status → level
- 错误字段：message/error/msg/detail → message
- 缺失字段：可推断则补全，否则保留空值并标记

**成功标准：**
- ✅ 字段名不一致时能正确映射
- ✅ 缺少关键字段不崩溃
- ✅ 脏数据原因被明确记录

---

### 3. 异常识别
**判断依据（满足任一即为异常）：**
1. level 包含：error/fatal/critical/failed/错误/失败
2. message 包含：timeout/exception/failed/denied/unavailable/超时/失败/异常/拒绝
3. status 表示失败
4. 字段缺失严重（影响业务判断）

**成功标准：**
- ✅ 正常日志不进入异常列表
- ✅ 异常日志能被正确识别
- ✅ 边界情况有明确处理规则

---

### 4. 异常分类
**分类体系：**
- `TIMEOUT` - 超时
- `AUTH` - 认证/权限问题
- `DATABASE` - 数据库问题
- `NETWORK` - 网络/依赖不可用
- `VALIDATION` - 参数/数据校验问题
- `RESOURCE_LIMIT` - 限流/容量/资源限制
- `DATA_QUALITY` - 脏数据/字段缺失
- `UNKNOWN` - 未知问题

**分类规则：** 基于关键词匹配 + 规则推理（不能写死结果）

**成功标准：**
- ✅ 分类结果可解释
- ✅ 不是简单的字符串映射表
- ✅ 有明确的分类推理逻辑

---

### 5. 异常合并（⭐ 核心竞争力）
**目标：** 将同一根因的不同日志合并为一个问题

**Fingerprint 生成算法：**
```
fingerprint = hash(
    标准化来源系统,
    标准化错误类型,
    归一化消息模板
)
```

**关键处理：**
1. **去除动态内容：**
   - 数字 ID（订单号、用户ID）
   - UUID/trace_id
   - 时间戳
   - IP 地址
   - 具体数值

2. **同义词归一化：**
   - timeout/timed out/超时 → TIMEOUT
   - failed/失败 → FAILED
   - permission denied/unauthorized/无权限 → AUTH
   - database/数据库/db → DATABASE

3. **中英文统一：**
   - 相同含义的中英文映射到同一标准词

**成功标准：**
- ✅ 含不同订单号的同一错误能合并
- ✅ 中英文同义词能归为同类
- ✅ 不同根因相似表述不会误合并
- ⚠️ 这是最难也最能体现水平的部分

---

### 6. 优先级判断（⭐ 核心竞争力）
**优先级定义：**
- `P0` - 核心链路大面积失败，立即处理
- `P1` - 重复发生或影响重要功能，尽快处理
- `P2` - 影响有限或可重试，正常排队
- `P3` - 低影响、偶发或可自动恢复

**评分算法（多因素加权）：**
```
score = 
    severity_weight × severity_score +       # 严重程度（30%）
    frequency_weight × log(1 + count) +      # 重复次数（25%）
    service_weight × is_core_service +       # 是否核心链路（20%）
    unknown_weight × is_unknown +            # 是否未知错误（15%）
    missing_field_penalty +                  # 缺少关键字段（10%）
    time_span_weight × time_continuity       # 是否持续发生
```

**核心服务定义：**
- payment（支付）
- order（订单）
- auth（认证）

**成功标准：**
- ✅ 不是简单的 level → priority 映射
- ✅ 重复次数增加后优先级升高
- ✅ 核心链路异常优先级高于非核心
- ✅ 优先级计算有明确公式

---

### 7. 人工介入判断
**需要人工介入：**
- UNKNOWN 类型异常
- P0/P1 优先级
- 同一问题重复次数 > 5
- 涉及核心业务链路
- 缺少关键字段无法确认根因
- 涉及权限/数据一致性/资金

**可自动处理：**
- P3 优先级
- 低频 timeout（次数 ≤ 2）
- 已知可重试任务
- 临时限流且影响小

**成功标准：**
- ✅ 判断依据明确
- ✅ UNKNOWN 默认需要人工介入
- ✅ 可自动处理的有合理建议

---

### 8. 处理建议
每个分类对应建议模板：
- **TIMEOUT：** 检查下游服务可用性、网络延迟和重试配置
- **AUTH：** 检查 token、权限配置或凭证是否过期
- **DATABASE：** 检查数据库连接池、慢查询、锁等待或实例状态
- **VALIDATION：** 检查输入数据格式、必填字段和上游数据质量
- **RESOURCE_LIMIT：** 检查限流配置、队列积压和容量水位
- **UNKNOWN：** 人工查看原始日志和上下文

---

### 9. 状态管理
**状态定义：**
- `open` - 待处理
- `investigating` - 排查中
- `mitigated` - 已缓解
- `resolved` - 已解决
- `ignored` - 已忽略

**操作：** 支持更新合并后问题的状态

---

### 10. 展示与统计
**异常列表展示：**
- 优先级、分类、来源、次数
- 首次/最后发生时间
- 是否需要人工介入
- 处理建议、当前状态

**统计信息：**
- 总日志数、有效数、脏数据数、异常数
- 合并后问题数
- 各优先级/分类分布
- 需要人工介入数量
- 按来源系统分布

## 📊 模拟数据要求

### 数量
至少 30 条，建议 45-60 条

### 复杂度覆盖
必须包含以下情况：

| 类型 | 样例 | 数量建议 |
|------|------|----------|
| 字段名不一致 | timestamp vs time vs created_at | 贯穿全部 |
| 同一任务重复失败 | 相同 job_name 多次出现 | 5-8 条 |
| 同一根因不同表述 | "DB timeout" / "数据库超时" | 6-10 条 |
| 时间乱序 | timestamp 不按顺序 | 随机分布 |
| 缺少关键字段 | 缺 timestamp 或 message | 3-5 条 |
| 无效 JSON | 解析失败的行 | 2-3 条 |
| 中英文混杂 | error 和错误混用 | 10+ 条 |
| 动态内容 | 含订单号、UUID、trace_id | 15+ 条 |

### 来源系统
至少 5 个不同系统：
- payment-service
- order-worker
- auth-gateway
- inventory-service
- notification-worker
